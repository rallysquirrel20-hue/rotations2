### Investigation: Parallel vs Batch Benchmark Calculation

**Problem**: After fixing the `incrementProgress` bug, benchmarks worked but took ~30s for a single-leg basket_tickers backtest (High Beta basket). The frontend was firing 7 parallel POST requests to `/api/backtest/multi` (1 main + 6 benchmarks).

**Testing methodology**: Created `bench_backtest.py` timing test script that measures:
- Single backtest latency
- 6 benchmarks sequential
- 6 benchmarks parallel
- 7 requests parallel (frontend pattern)
- New batch endpoint

**Key finding — parallel requests do NOT help**:
- Single backtest: ~4.5s
- 6 benchmarks sequential: ~29.7s (sum of ~4.9s each)
- 6 benchmarks parallel: ~26.6s (only 1.1x speedup — nearly sequential!)
- 7 parallel (old frontend): ~29.6s

**Root cause**: Python GIL. FastAPI dispatches sync `def` endpoints to a threadpool, but CPU-bound pandas/numpy work (parquet reads, DataFrame operations, trade building) holds the GIL. 7 threads all competing for the same GIL = effectively sequential execution.

**Solution that worked**: New `/api/backtest/benchmarks` batch endpoint that:
1. Loads the parquet ONCE (the expensive ~2s I/O operation)
2. Resolves universe tickers ONCE
3. Builds ticker_closes dict ONCE
4. Loops over 6 signals sequentially for trade extraction + equity replay
5. Returns all 6 equity curves + stats in one response

This eliminates 6 redundant parquet loads, 6 redundant universe resolutions, and all GIL contention. Total time dropped from ~30s to ~10-12s.

**Lesson**: For CPU-bound Python backends, parallel HTTP requests are worse than a single batched request. The GIL serializes the work anyway, and each request adds overhead (parquet reload, universe lookup, HTTP round-trip).
