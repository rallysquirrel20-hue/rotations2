## Architecture decisions (locked in during design)

- **Yield definition**: `TTM_Dividends / Close` using Norgate CAPITALSPECIAL-adjusted prices (split-adjusted, not dividend-adjusted). Single Norgate call per ticker. Yield stored as decimal (0.025 = 2.5%).
- **Non-payer convention**: yield = 0 AND growth = 0 whenever TTM is 0 — never NaN / Inf. Applies at ticker level and propagates through basket aggregation as weighted zeros. **No coverage threshold, no graying-out, no hiding.** User explicitly chose this: "If a ticker has no dividend, it has a 0% growth rate." Saved to project memory.
- **Basket weighting**: weighted average using the existing drifted `Weight_BOD` column from `{slug}_*_contributions.parquet`. No new weight computation — reuses the dollar-volume-at-quarter-start weights that drift via cumprod through the quarter.
- **TTM/YoY window**: started at 252 trading days (matching legacy `_build_dividends`), switched to 365 calendar days mid-session after the user observed multi-day spikes around SPY's quarterly ex-div dates. Calendar-aligned windows reduce spikes to 1 trading day (the unavoidable minimum absent a fixed-N-event approach).

## Changes

### 1. `signals/build_dividend_metrics.py` — new producer module
One Norgate call per ticker (CAPITALSPECIAL price series). Computes TTM as `divs.rolling('365D').sum()` on a DatetimeIndex, yield as `TTM / Close`, YoY via `ttm.asof(t - 365D)`. Full rebuild only (incremental intentionally deferred — 365-day YoY lag would have to recompute a full year anyway). Runs via `python build_dividend_metrics.py --force`. Full 2554-ticker equity + 224-ticker ETF build = ~20s wall time.

### 2. `signals/config.py` — new constants
- `DIVIDEND_METRICS_CACHE_FILE`, `DIVIDEND_METRICS_ETF_CACHE_FILE` — parquet paths
- `DIVIDEND_METRICS_SCHEMA_VERSION = 2` (bumped from 1 on calendar-day switch)
- `DIVIDEND_TTM_WINDOW = 365`, `DIVIDEND_YOY_LAG = 365` — **calendar** days (semantics shift from trading days)

### 3. `signals/loop_signals.py` — wired in
New script inserted between `build_signals.py` and `build_baskets.py`. Sequential — baskets need fresh per-ticker metrics before the aggregation post-pass runs.

### 4. `signals/build_universes.py:_build_dividends` — window-consistency update
`calc_trailing_divs` switched from `.rolling(252, min_periods=1).sum()` to `.rolling('365D').sum()` so universe selection uses the same TTM definition as the displayed metrics. No rebuild triggered — current quarterly dividend universes stand. Next rebuild at end of Q2 (2026-06-30) will pick up the new code path automatically.

### 5. `signals/build_baskets.py` — basket aggregation post-pass
Two new functions:
- `_compute_basket_dividend_series(contrib_df, div_metrics)` — weighted yield, weighted growth, payer coverage per date. Non-payers contribute 0 to both.
- `_build_all_basket_dividend_series()` — post-pass called from `main()` after the basket loop. Globs every `*_contributions.parquet` across thematic/sector/industry cache folders, joins with `dividend_metrics_500.parquet` (pre-indexed by ticker for fast per-basket filtering), writes matching `*_dividend_series.parquet`.

Wall time: 75 basket series written in ~15s. Well under the <10min full-rebuild budget.

### 9. Calendar-day TTM/YoY switch (late-session fix)
User observed multi-day yield/growth spikes on SPY around quarterly ex-div dates. Root cause: 252 trading days ≈ 366 calendar days **on average** but drifts between 363-368 day-to-day. Ex-div dates are calendar-spaced, so the rolling boundary sometimes caught 5 distributions in the window and sometimes 3. Positional (trading-day) rolling had 5-7 day spikes per ex-div; switching to time-based 365-day rolling narrowed them to 1 trading day (the unavoidable minimum without a fixed-N-event window). Yahoo/Bloomberg's published TTM yields exhibit the same 1-day jump — it's a fundamental property of TTM aggregation.

Switched both `build_dividend_metrics.py` (daily series) and `build_universes.py` (quarter-end snapshot used for `Dividend_Growth` universe selection). Rebuilt all parquets:
- `dividend_metrics_500.parquet` — 18s
- `dividend_metrics_etf_50.parquet` — 1.8s
- 75 basket `*_dividend_series.parquet` via post-pass — 15s
